2. CONTROLADOR
========================================

Archivo: src/Modules/Simulator/Controllers/PdfReportController.php

Método:  generar()

Flujo:
  1. Obtiene el estudiante_id desde la sesión (user_id → StudentAssignmentModel)
  2. Obtiene el intento activo (sim_asignacion_id → StudentAttemptModel)
  3. Instancia DeclaracionComparador y llama comparar(intentoId, estudianteId)

  6. Genera PDF con mPDF (Letter, UTF-8, dejavusans) y lo streaming inline

3. SERVICIO PRINCIPAL: DeclaracionComparador
========================================

Archivo: src/Modules/Simulator/Services/DeclaracionComparador.php

Método público: comparar(intentoId, estudianteId) → array

FUENTES DE DATOS:

  - UnidadTributariaService: obtiene el valor de la UT por fecha

========================================
4. SECCIONES COMPARADAS (en orden)
========================================

El método comparar() construye un array $secciones donde cada sección tiene:
  { titulo: string, grupos: array }

Cada grupo tiene:
  { label: string, campos: array }

Cada campo tiene:
  { campo: string, borrador: string, esperado: string, correcto: bool, tipo?: string }

Los tipos especiales de campo son: 'omitido', 'sobrante' (de más).

SECCIONES:
──────────────────────────────────────────
4.1  HEREDEROS REGULARES
     Método: compararHerederosRegulares()
     Matching: por documento (cédula/RIF/pasaporte)
     Campos por heredero: Nombre, Documento, Parentesco, Premuerto,
                          Fecha Nacimiento, Fecha Fallecimiento (si premuerto)
     Grupo especial: "Cantidad" (resumen, no cuenta como unidad)

4.2  HEREDEROS DEL PREMUERTO
     Método: compararHerederosDelPremuerto()
     Matching: por documento
     Campos: Nombre, Documento, Parentesco, Fecha Nacimiento, Representa a

4.3  PRÓRROGAS
     Método: compararProrrogas()
     Matching: score-based (nro_resolucion +2, fecha_solicitud +1 — 2 pasadas)
     Campos: Fecha Solicitud, Nro. Resolución, Fecha Resolución,
             Plazo Otorgado (Días), Fecha Vencimiento

4.4  TIPO DE HERENCIA
     Método: compararTipoHerencia()
     Matching: por tipo_herencia_id (intersección/diferencia de IDs)
     Campos: Tipo, Subtipo Testamento, Fecha Testamento,
             Fecha Conclusión Inventario

4.5  BIENES INMUEBLES
     Método: compararBienesInmuebles()
     Matching: score-based (descripcion +2, valor +1 — 2 pasadas)
     Campos: Valor Declarado, Porcentaje, Descripción, Vivienda Principal,
             Bien Litigioso + datos tribunal si es litigioso,
             Oficina Registro, Nro Registro, Libro, Protocolo, Trimestre,
             Fecha Registro, Superficie Construida/No Construida, Valor Original

4.6  BIENES MUEBLES (12 sub-secciones independientes)
     Método: compararBienesMuebles()
     Categorías: Banco, Seguro, Transporte, Opciones de Compra,
                 Cuentas y Efectos, Semovientes, Bonos, Acciones,
                 Prestaciones Sociales, Caja de Ahorro, Plantaciones, Otros
     Matching: score-based (tipo_bien +3, descripcion +2, valor +1)
     Campos: Tipo de Bien (si aplica), Valor Declarado, Porcentaje, Descripción,
             Bien Litigioso + datos tribunal
     Cada categoría genera su propia sección: "Bienes Muebles — [Categoría]"

4.7  PASIVOS DEUDA (4 sub-secciones independientes)
     Método: compararPasivosDeuda()
     Sub-tipos: TDC, Crédito Hipotecario, Préstamos Cuentas y Efectos, Otros
     Matching: score-based genérico (descripcion +2, valor +1)
     Campos: Valor Declarado, Porcentaje, Descripción
     Cada sub-tipo genera su propia sección: "Pasivos Deuda — [Tipo]"

4.8  PASIVOS GASTOS
     Método: compararPasivosGastos()
     Matching: score-based genérico
     Campos: Valor Declarado, Porcentaje, Descripción

4.9  EXENCIONES
     Método: compararExclusion() — lógica propia (NO usa compararItemsGenerico)
     Tabla DB: sim_caso_exenciones
     Matching: score-based (tipo +3, descripcion +2, valor +1)
     Campos: Tipo, Descripción, Valor Declarado (sin porcentaje)

4.10 EXONERACIONES
     Método: compararExclusion() — misma lógica que exenciones
     Tabla DB: sim_caso_exoneraciones
     Matching: score-based (tipo +3, descripcion +2, valor +1)
     Campos: Tipo, Descripción, Valor Declarado (sin porcentaje)

──────────────────────────────────────────

========================================
5. AUTOLIQUIDACIÓN (14 líneas)
========================================

Método: compararAutoliquidacion()
No forma parte de $secciones — es un array independiente de 14 items.

Líneas comparadas:
  1.  Total Bienes Inmuebles
  2.  Total Bienes Muebles
  3.  Patrimonio Hereditario Bruto (1+2)
  4.  Activo Hereditario Bruto
  5.  Desgravámenes
  6.  Exenciones
  7.  Exoneraciones
  8.  Total Exclusiones (5+6+7)
  9.  Activo Hereditario Neto (4-8)
  10. Total Pasivo
  11. Patrimonio Neto Hereditario (9-10)
  12. Impuesto Determinado según Tarifa (vía TributoCalculator)
  13. Reducciones (vía TributoCalculator)
  14. Total Impuesto a Pagar (vía TributoCalculator)

Fuentes:
  - Borrador: BorradorService calcula totales desde el JSON
  - DB: sumFromTable() y calcDesgravamenesDb() sobre las tablas del caso

========================================
6. CÁLCULO POR HEREDERO
========================================

Método: compararCalculoHerederos()
No forma parte de $secciones — es un array independiente.

Por cada heredero del borrador:
  Campos: Cuota Parte (UT), Porcentaje (%), Sustraendo (UT),
          Impuesto Determinado (Bs), Reducción (Bs), Impuesto a Pagar (Bs)

  Notas diagnósticas contextuales (array 'notas' por heredero):
    - Escenario B: PN ≤ 0 + parentesco incorrecto
      → "El parentesco declarado (X) no coincide con el correcto (Y).
         Además, el PN correcto es ≤ 0, no hay impuesto independientemente."
    - Escenario D: PN > 0 + parentesco incorrecto
      → "El parentesco declarado (X) no coincide con el correcto (Y).
         Esto afecta el grupo de tarifa, porcentaje, sustraendo e impuesto."
    - Escenario E: PN > 0, parentesco OK, cuota parte incorrecta
      → "La Cuota Parte difiere porque el PN declarado (X Bs) no coincide
         con el correcto (Y Bs), o la cantidad de herederos difiere."
    - Escenario F: Heredero exento Art. 9 pero estudiante calculó impuesto
      → "Este heredero está exento según Art. 9 (Cuota ≤ 75 UT + G1).
         Impuesto correcto: 0,00 Bs."

Cálculo vía TributoCalculator:
  1. Cuota parte = patrimonio_neto / total_herederos (en UT)
  2. Lookup de grupo tarifario: parentesco_id → grupo (1-4)
  3. Lookup de tramo: sim_cat_tarifas_sucesion por grupo + cuota
  4. Art. 9: si grupo 1 y cuota ≤ 75 UT → impuesto = 0
  5. Fórmula: impuesto_ut = (cuota_ut × % / 100) − sustraendo_ut
  6. Impuesto Bs = impuesto_ut × valor_ut

========================================
7. SCORE GLOBAL
========================================

Calculado en comparar() después de construir todas las secciones.

Criterio por sección:
  - Cada "grupo" = 1 unidad evaluada (excepto label "Cantidad")
  - Si label empieza con "Omitido:" → omitida
  - Si label empieza con "De más:" → de_mas
  - Si todos los campos son correcto=true → correcta
  - Si al menos un campo es correcto=false → con_errores

Resultado: { correctas, con_errores, omitidas, de_mas, total_esperado, porcentaje }
  donde total_esperado = correctas + con_errores + omitidas
  y porcentaje = (correctas / total_esperado) × 100

Autoliquidación: cada línea = 1 unidad (correcto o con_errores)
Herederos Calculo: cada heredero = 1 unidad (todos campos OK o no)

========================================
10. ALGORITMO DE MATCHING (2 PASADAS)
========================================

Usado en: Inmuebles, Muebles, Pasivos Deuda/Gastos, Prórrogas,
          Exenciones, Exoneraciones

PASADA 1 — Score-based:
  Para cada ítem del borrador, calcular un score contra cada ítem DB:
    Criterios (varían por sección):
      tipo_bien coincide    +3
      tipo coincide         +3
      nro_resolucion        +2
      descripcion coincide  +2
      fecha_solicitud       +1
      valor_declarado       +1
  Emparejar con el mejor score > 0 (greedy, primer match gana)

PASADA 2 — Ordinal fallback:
  Los ítems sin match de ambos lados se emparejan por orden posicional

SOBRANTES: ítems del borrador que no se pudieron emparejar (de más)
FALTANTES: ítems de la DB que no se emparejaron (omitidos)
